# KoreanKids — Engineering Blueprint v2
## 유지비 $0 · 1인 개발 · 한국 문화유산 테마

> **v2 주요 변경사항**
> 1. 인프라 유지비를 **$0/월** 제약 아래에서 재설계
> 2. 한국어 교육 전문가 외주 **제외** (AI + 공공자료 기반 대체 전략)
> 3. 한국 문화유산 테마를 각 Learning Stage에 매핑 (공공누리 활용)
> 4. Claude Design을 프로덕션 디자인 툴로 공식 채택
>
> **문서 버전**: v2.0
> **작성 기준일**: 2026년 4월
> **v1과의 관계**: v1의 기본 원칙과 아키텍처는 유지. 제약 조건(비용·인력·콘텐츠)이 바뀌면서 구현 방식이 달라지는 부분을 재설계.

## 0. v2 설계의 대전제

**제약이 설계를 결정한다.** 이번에 추가된 세 가지 제약은 단순한 절약이 아니라 제품 전략 자체를 재정의한다:

| 제약 | 함의 |
|---|---|
| 유지비 $0/월 | 클라우드 DB·유료 호스팅 금지. 엣지·정적·클라이언트 중심 아키텍처. |
| 교육 전문가 없음 | 커리큘럼을 **공공 교재 + LLM + 오픈소스 데이터**로 자체 제작. 검증은 베타 사용자·학부모로 대체. |
| 한국 문화유산 테마 | 단순 학습 앱이 아니라 **"문화 여행"** 브랜드 포지셔닝. 차별화 축 하나가 더 생김. |

이 세 제약은 서로 시너지가 있다:
- 공공누리 문화유산 자료 → 무료로 쓸 수 있는 풍부한 시각 자산 → 일러스트 외주 불필요
- 공공 교재(국립국어원·세종학당) → 무료 커리큘럼 원천 → 전문가 외주 불필요
- Claude Design → UI 디자인 외주 불필요
- 모든 게 Claude 생태계에서 해결 → 유지비도 한 번 내고 있는 구독료 내에서 해결

즉 **"공공 영역(public commons) + AI 생태계"만으로 제품을 완성**하는 전략이다.

## 1. 유지비 $0 아키텍처 (Zero-Cost Stack)

### 1.1 핵심 원칙

- **정적(static)이 동적(dynamic)보다 저렴하다** → 가능한 한 모든 콘텐츠를 정적 자산으로
- **클라이언트가 서버보다 저렴하다** → 로직을 클라이언트로 밀어넣는다
- **CDN은 사실상 공짜다** → CDN이 DB 역할까지 하게 한다
- **"무료 티어"로 시작하고 그 안에 머무른다** → 초기 사용자 1,000명까지는 $0 가능

### 1.2 기술 스택 (Zero-Cost)

| 레이어 | 선택 | 무료 한도 (MVP 기준 충분) |
|---|---|---|
| **모바일 앱** | Expo React Native + EAS | 빌드 월 30회 무료, OTA 업데이트 무제한 |
| **웹 (교사·부모)** | Next.js + Cloudflare Pages | 무제한 요청, 500 빌드/월 |
| **정적 콘텐츠 호스팅** | Cloudflare R2 + Pages | 10GB 스토리지, 무료 egress |
| **데이터베이스** | **Cloudflare D1** (SQLite 엣지) | 읽기 500만/일, 쓰기 10만/일 |
| **인증** | Clerk 무료 플랜 (10K MAU) 또는 자체 구현 (D1 기반) | 10,000 MAU까지 무료 |
| **백엔드 로직** | Cloudflare Workers | 10만 요청/일 무료 |
| **파일 스토리지** | Cloudflare R2 | 10GB 무료, egress 무료 |
| **이메일 발송** | Resend 무료 (3,000/월) | MVP 충분 |
| **분석** | Cloudflare Web Analytics (무료) + 자체 이벤트 로그 → D1 | 무제한 |
| **에러 추적** | Sentry 무료 티어 (5K events/월) | MVP 충분 |
| **CI/CD** | GitHub Actions | public repo 무제한, private 2,000분/월 |
| **도메인** | Cloudflare Registrar (원가) | 연 $10 정도 — 유일한 실비 |

**실제 월 비용**: **$0** (도메인 연 $10 제외)

### 1.3 왜 Cloudflare 중심인가

Supabase, Railway, Vercel 모두 좋은 옵션이지만 **무료 티어의 제한 조건**이 MVP 이후 걱정되는 구조:
- Supabase: DB 프로젝트가 7일간 미사용 시 자동 일시정지 (pause) — 신규 사용자 UX 망침
- Railway: $5 크레딧 이후 유료
- Vercel: 상업 용도면 Pro 전환 압박

**Cloudflare는 "free tier가 진짜로 풀려있다"** — 10만 요청/일이 MVP에서 MAU 수천 명까지 커버 가능. 제한에 가까워지면 워커당 $5/월로 확장 (무료와 유료 사이 절벽이 없음).

### 1.4 아키텍처 다이어그램 (Zero-Cost)

```
[모바일 앱 (Expo)]          [웹 (Next.js)]
       │                         │
       └──────────┬──────────────┘
                  ▼
       ┌────────────────────┐
       │  Cloudflare Edge   │
       │  (CDN + Workers)   │
       └─────────┬──────────┘
                 │
     ┌───────────┼──────────────┐
     ▼           ▼              ▼
┌─────────┐ ┌─────────┐  ┌─────────────┐
│   R2    │ │   D1    │  │ Auth (Clerk │
│  (자산)  │ │ (SQLite)│  │  또는 자체)  │
└─────────┘ └─────────┘  └─────────────┘
     │
     ▼
[Static content: lessons.json, audio/, images/]
```

### 1.5 $0 유지의 트레이드오프

| 장점 | 단점 |
|---|---|
| 재정적 zero-risk로 장기 운영 가능 | 동시성·쓰기 많은 기능 설계 제한 |
| 스케일 자동 (Cloudflare edge) | SQL 기능 제한 (D1은 full Postgres 아님) |
| DDoS 방어 무료 포함 | 특정 지역 지연 가능 (드물게) |
| 개발자 생산성 높음 (Wrangler CLI) | Cloudflare 생태계 락인 |

결론: MVP~MAU 10,000명까진 무료. 이후 필요시 Cloudflare Paid 또는 부분 마이그레이션.

## 2. 교육 전문가 없이 커리큘럼 만드는 법

### 2.1 핵심 전략: **공공 교재 + Claude + 베타 검증**

전문가 없이 커리큘럼을 만드는 건 위험할 수 있지만, 한국어 교육은 이미 **국가 차원에서 고품질 공공 교재가 풍부**하다. 이를 소스로 활용하면 '전문가가 설계한 커리큘럼'을 간접적으로 재현할 수 있다.

### 2.2 활용할 무료 공공 자원

| 자원 | 제공 기관 | 라이선스 | 활용 방식 |
|---|---|---|---|
| **초등학생을 위한 표준 한국어 (KSL) 교재** | 국립국어원 | 공공누리 (상당수 제1유형) | 커리큘럼 뼈대·어휘·문형 |
| **세종한국어** | 세종학당재단 | 학습자 무료 공개 | 레벨별 문형·어휘 |
| **다문화가정 유아 활동지 270개** | 국립국어원 | 공공누리 | Level 0 미니게임 문항 |
| **한국어기초사전** | 국립국어원 | 공공누리 | 단어 정의·예문 |
| **훈민정음 해례본** | 간송미술문화재단 | 저작권 소멸 | Level 0 스토리 소스 |
| **한국민족문화대백과사전** | 한국학중앙연구원 | 공공누리 (글 자유이용) | 문화 설명 |
| **공공누리 누리집 저작물 500만 건+** | 다수 공공기관 | 유형별 상이 | 시각 자산 |

### 2.3 커리큘럼 생성 파이프라인 (LLM 활용)

```
[1] 공공 교재에서 각 Level별 교육 목표 추출
    예: KSL 교재 1권 = 한글 자모 → 기본 어휘 80 → 인사말
         → 우리 앱 Level 0~1 대응

[2] Claude에 구조화된 프롬프트로 레슨 초안 생성
    - 입력: 교육 목표, 어휘 리스트, 문형
    - 출력: 우리 앱의 Lesson JSON (Hook/Discover/Play/Check/Celebrate)

[3] 두 번째 Claude 패스로 검증
    - 난이도 적절성
    - 문화적 편향 (한 국가만 미화 등)
    - 연령 적합성 (5~11세)

[4] 학부모 10명 베타 리뷰
    - "우리 아이가 이걸로 배울 수 있을까?"
    - 불명확한 부분만 수정

[5] 콘텐츠 버전 관리 (Git)
    - 문제 발생 시 빠른 롤백
```

### 2.4 검증을 전문가에서 사용자로 이동

**전통적 방식**: 전문가가 설계 → 개발 → 출시
**우리 방식**: LLM + 공공 교재로 초안 → **베타 가정 10~20 조기 투입** → 피드백으로 반복 수정

이는 오히려 실제 사용자와 밀착된 제품을 만들 가능성이 있다. 단, 몇 가지 리스크 관리가 필요:

### 2.5 전문가 부재의 리스크 완화

| 리스크 | 완화 방식 |
|---|---|
| 발음 부정확 | 원어민 녹음은 타협 불가. 하지만 TTS 품질이 이미 준수 — 네이버 Clova Dubbing 등 저비용 활용. 최후엔 본인 가족·친구 녹음으로도 MVP 가능 |
| 학습 순서 비효율 | 공공 교재가 이미 검증된 순서 제공 → 그대로 따르기 |
| 문화 맥락 오류 | 본인이 한국인이므로 일부 보완 가능 + 베타 가정 피드백 |
| 장애 아동 접근성 무지 | MVP에선 정상 발달 아동 집중, Phase 2에서 접근성 확장 |
| 학부모 신뢰도 | "국립국어원 KSL 교재 기반"을 명시적으로 표기 → 권위 이식 |

### 2.6 프롬프트 예시 (Claude로 레슨 생성)

```
당신은 6~8세 어린이를 위한 한국어 학습 앱의 커리큘럼 설계자입니다.
아래의 국립국어원 KSL 교재 단원 내용을 바탕으로 우리 앱의 Lesson JSON을 생성해주세요.

[원본 교재 단원]
...

[우리 앱의 Lesson 구조]
- Hook (30s, 캐릭터 호야가 인사)
- Discover (90s, 새 내용 소개)
- Play (150s, 미니게임 2~3개)
- Check (60s, 3문항 퀴즈)
- Celebrate (30s, 보상)

[제약]
- 각 레슨 총 6분 이내
- 영어 UI 기준 설명 최소화
- 한국 문화유산 테마 [이번 레슨은 "훈민정음"] 과 연계
- 아동 정서적으로 부정적 표현 금지

JSON 스키마: [...]
```

## 3. 한국 문화유산 테마 — "Heritage Journey" 브랜드

### 3.1 새로운 브랜드 중심 설정

이 테마 결정은 부가 요소가 아니라 **제품의 중심 서사**가 될 수 있다. 어린이에게 한국어를 가르치는 수많은 앱 중에서 **"한국 문화유산을 모험처럼 탐험하며 한국어를 배우는 앱"** 은 포지셔닝이 완전히 새롭다.

### 3.2 Learning Stage × 문화유산 매핑

각 Level을 한국 문화유산 시대·장소와 연결:

```
┌──────────────────────────────────────────────────────────────┐
│ Level 0: Hangul Discovery                                    │
│ 테마: 🏛 집현전 & 훈민정음 해례본                              │
│ 시대: 1443년, 세종의 서재                                     │
│ 캐릭터: 세종대왕의 고양이 "훈이" (창작) + 집현전 학자들       │
│ 서사: 호야가 집현전에 시간여행 → 한글 창제 과정을 함께 경험    │
│ 시각 자산: 훈민정음 해례본 (저작권 소멸, 자유 사용)           │
│ 미니게임: 자음은 조음기관 모양 / 모음은 천지인에서 유래        │
├──────────────────────────────────────────────────────────────┤
│ Level 1: First Words                                         │
│ 테마: 🌿 승정원일기 속 조선 생활                               │
│ 시대: 조선 궁궐·시장·가정                                     │
│ 캐릭터: 궁궐 아이·시장 상인·농부 등                            │
│ 서사: 조선 시대 하루를 따라가며 생활 단어 습득                 │
│ 시각 자산: 김홍도 풍속화 (국립중앙박물관 공공누리 제1유형)     │
│ 미니게임: 씨름도·서당도 속 인물들과 대화                       │
├──────────────────────────────────────────────────────────────┤
│ Level 2: Sentence Patterns                                   │
│ 테마: 📜 팔만대장경 & 도서관 모험                              │
│ 시대: 고려·조선 책 문화 (경판·활자·한지)                       │
│ 캐릭터: 책 요정 "활이" — 글자 조각들로 문장을 만드는 존재      │
│ 서사: 흩어진 경판 조각들을 맞추며 문장 패턴 익힘               │
│ 시각 자산: 팔만대장경 경판 사진 (문화재청 공공누리)            │
│ 미니게임: 경판 퍼즐, 활자 조립                                 │
├──────────────────────────────────────────────────────────────┤
│ Level 3: Conversations (킬러 기능)                            │
│ 테마: 🎭 종묘 제례 & 판소리                                    │
│ 시대: 전통 의례·공연 문화                                      │
│ 캐릭터: 소리꾼 할아버지, 어린 궁중무용수                        │
│ 서사: 여러 전통 의례·축제에서 역할을 맡아 대화                 │
│ 시각 자산: 종묘·무형문화재 사진 (국가유산청 공공누리)          │
│ 음성 인식 기반 역할극                                          │
├──────────────────────────────────────────────────────────────┤
│ Level 4: Read & Write                                        │
│ 테마: 📚 조선왕조실록 & 일기 문화                              │
│ 시대: 기록 문화 — 사관·일기·편지                               │
│ 캐릭터: 어린 사관 "기록이"                                      │
│ 서사: 자기만의 작은 실록·일기·편지를 써보는 활동               │
│ 시각 자산: 실록 이미지·옛 편지 (국사편찬위원회 공공누리)        │
│ Final Cert: 조선왕조실록 형식의 개인 "나의 실록" 발급          │
└──────────────────────────────────────────────────────────────┘
```

### 3.3 이 접근의 교육적 가치

단순 한국어 학습 이상의 가치:
- **메타 학습**: "한글은 왜 이렇게 생겼을까?" → 발음 원리 이해 → 빠른 습득
- **문화 자본**: 재외동포 2·3세에게 조상의 역사 접속
- **글로벌 시장성**: K-컬처 관심자에게는 "진짜 한국"을 만나는 경험
- **세계 기록유산 연계**: 훈민정음·조선왕조실록·팔만대장경이 이미 유네스코 세계기록유산

### 3.4 차별화 관점의 새로운 포지셔닝

```
                  (문화 맥락 깊이)
                         │
                         │
                         │  ★ 우리 (KoreanKids Heritage)
                         │  · 문화유산 여행 = 학습
                         │  · 단순 어휘 앱이 아님
                         │  · 부모·교사가 "의미 있는" 앱으로 인식
                         │
   ─────────────────────┼──────────────────────
```

## 4. 공공누리 & 문화유산 저작권 — 실무 가이드

### 4.1 한국의 공공저작물 이용 체계

한국은 **공공누리(KOGL)** 라는 표준화된 공공저작물 이용허락 체계를 운영 중이다. 저작권법 제24조의2(공공저작물의 자유이용)에 따라 국가·지방자치단체·공공기관이 저작재산권을 보유한 저작물은 공공누리 유형에 따라 자유롭게 이용 가능하다.

### 4.2 공공누리 4개 유형 핵심 요약

| 유형 | 출처 표시 | 상업적 이용 | 변경 (2차 저작) |
|---|---|---|---|
| **제1유형** | 필수 | ✅ 허용 | ✅ 허용 |
| **제2유형** | 필수 | ❌ 금지 | ✅ 허용 |
| **제3유형** | 필수 | ✅ 허용 | ❌ 금지 |
| **제4유형** | 필수 | ❌ 금지 | ❌ 금지 |

제1유형은 이용자가 출처 표시만 하면 상업적 활용 여부에 관계없이 무료로 자유롭게 이용하고 2차적 저작물 작성 등 변형이 가능하다. 우리 앱은 **제1유형만 채택**하는 것이 안전.

### 4.3 핵심 판단: 우리 앱에 쓸 수 있나?

| 자원 | 우리 앱 활용 가능성 | 근거 |
|---|---|---|
| **훈민정음 해례본** | ✅ **완전 자유** | 1446년 제작 → 저작권 보호기간 만료 (사후 70년 훨씬 지남). 보호기간이 만료된 저작물은 공공누리 부착 여부와 무관하게 자유 이용 가능. |
| **조선왕조실록** | ✅ **자유** | 저작권 보호기간 만료. 단 국사편찬위 촬영 사진은 해당 기관 공공누리 유형 확인 필요 |
| **팔만대장경** | ✅ **자유** | 저작권 보호기간 만료 |
| **종묘·성균관 등 건축물** | ✅ **자유** | 건축물 자체는 저작권 대상 아님. 사진은 촬영자 저작권 (단 공공기관 사진은 공공누리 해당) |
| **김홍도 풍속화** | ✅ **자유** | 김홍도의 '씨름'은 공공저작물로 지정되어 출처만 밝히면 상업적 용도로도 자유롭게 사용 가능. |
| **국립중앙박물관 소장품 이미지** | ⚠️ **확인 필요** | 공공누리 제1~4유형이 혼재. 개별 저작물마다 유형 확인 후 상업적 이용·변경 가능 여부 판단. |
| **국가유산청 (구 문화재청) 사진** | ⚠️ **확인 필요** | 공공누리 유형 표시를 확인해야 하며, 제2·4유형은 상업적 이용 금지, 제3·4유형은 변경 금지. |
| **한국민족문화대백과사전 글** | ✅ **자유** | 한국민족문화대백과사전은 공공저작물로서 공공누리 제도에 따라 이용 가능. 단 글 인용 시 '[출처 : 항목명 - 한국민족문화대백과사전]' 출처 표기 필수. 단 미디어 자료는 개별 확인 |

### 4.4 실무 체크리스트

모든 이미지·자료 사용 전 다음 체크:

```
□ 1. 저작물에 공공누리 마크가 표시되어 있는가?
□ 2. 표시가 없다면 → 제작자 사후 70년 경과 or 공표 후 70년 경과했는가?
□ 3. 공공누리 유형이 상업적 이용을 허용하는가? (제1·3유형만 OK)
□ 4. 공공누리 유형이 변경·2차 저작을 허용하는가? (제1·2유형만 OK)
□ 5. 출처 표시 문구를 생성·기록했는가?
□ 6. 사용 내역을 프로젝트 파일(assets-license.csv)에 기록했는가?
```

**결론**:
- **상업적 서비스인 우리 앱은 제1유형만 쓰는 게 안전함**. 변경·2차 저작 금지 조항이 걸리면 앱 내 리사이즈·크롭·색조 조정도 이슈가 될 수 있음.
- 저작권 소멸된 역사 자료 (훈민정음·실록·팔만대장경·김홍도)는 **가장 자유롭고 안전한 선택**.

### 4.5 추가 주의사항

공공기관이 후원한다거나 공공기관과 특수한 관계에 있는 것처럼 제3자가 오인하게 하는 표시는 금지된다. 앱 설명에서 "국립국어원 공식 앱" 같은 오해를 살 표현은 쓰면 안 됨. "국립국어원 KSL 교재 기반"은 OK.

### 4.6 출처 표시 구현 방법

앱 내 [About → Credits] 화면에 자동 생성된 표를 두고, 각 레슨에서 쓴 이미지·글의 공공누리 출처를 한 번에 고지. 개별 화면마다 출처 넣으면 어린이 UX 해침.

```
[Credits 화면 예시]
본 앱은 다음 공공저작물을 활용하였습니다:

• 김홍도 풍속도첩 — 국립중앙박물관 (공공누리 제1유형)
• 훈민정음 해례본 — 저작권 보호기간 만료
• 조선왕조실록 이미지 — 국사편찬위원회 (공공누리 제1유형)
...

전체 목록: [링크]
```

### 4.7 해외 자료 활용 주의

미국·일본 등 해외 박물관·기관의 한국 문화유산 이미지는 각 국가 저작권법 적용. 일괄 "공공자료"가 아님. **국내 기관 자료에 집중**하는 것이 안전.

## 5. Claude Design 채택 — 실제 워크플로우

### 5.1 Claude Design의 확인된 기능

2026년 4월 17일 출시된 Claude Design은 Anthropic Labs 제품으로, Opus 4.7 기반이며 Claude Pro·Max·Team·Enterprise 구독자에게 리서치 프리뷰로 제공된다. 프로토타입·와이어프레임·슬라이드·원페이저 등 완성도 있는 비주얼 작업을 대화로 만들 수 있다.

핵심 기능:
- 온보딩 시 팀의 코드베이스와 디자인 파일을 읽어 디자인 시스템을 구축하고, 이후 모든 프로젝트에 색상·타이포그래피·컴포넌트를 자동 적용한다. 디자인 시스템은 시간에 따라 다듬을 수 있고, 팀이 여러 개를 유지할 수 있다.
- 텍스트 프롬프트, 이미지·문서 업로드(DOCX·PPTX·XLSX), 코드베이스 포인팅으로 시작 가능. 웹 캡처 도구로 웹사이트 요소를 직접 가져올 수도 있다.
- 인라인 댓글·직접 편집·조절 슬라이더로 세밀 조정 가능. 조직 단위 공유 지원.
- 조직 내 URL·폴더 저장·Canva·PDF·PPTX·독립 HTML 파일로 내보내기. 빌드 준비되면 Claude Code로 핸드오프 번들 생성.

### 5.2 우리 프로젝트에 이상적인 이유

1. **Figma 라이선스·학습 불필요** — 1인 개발자에게 최적
2. **Claude Code와 네이티브 연결** — 디자인→구현이 원 워크플로우
3. **디자인 시스템 자동 적용** — 일관성을 AI가 보장
4. **구독료 내 포함** — Claude Design 사용은 기존 Claude 채팅·Claude Code 한도와 별도로 추적되며, 구독 플랜에는 주간 할당량이 별도로 제공된다. 추가 비용 없음
5. **Canva 내보내기** — 필요시 팀원·외주자와 협업 가능

### 5.3 우리 디자인 워크플로우

```
[1] 디자인 시스템 초기 구축
    Claude Design에 우리 프로젝트 소개 + 한국 전통 모티프·색상 가이드
    → 디자인 토큰 자동 생성 (colors, typography, spacing, components)

[2] Heritage 테마별 시각 기조 확정
    · Level 0: 집현전 (푸른 먹·한지색·단청 강조색)
    · Level 1: 풍속화 (따뜻한 세피아·붉은 도장)
    · Level 2: 활자 (먹색·금박)
    · Level 3: 제례 (붉은색·금색)
    · Level 4: 기록 (미색 한지·먹선)

[3] 화면 단위 디자인 생성
    "온보딩 첫 화면, 호야가 집현전 배경에서 인사하는 장면"
    → Claude Design이 초안 생성 → 조정

[4] Claude Code로 핸드오프
    완성 디자인 → React Native 코드 번들
    → GitHub에 PR

[5] 이후 수정은 Claude Design에서 대화로
```

### 5.4 이미지 에셋 통합 흐름

```
[공공누리 자료 수집]
    ↓
[Claude에 업로드 + 저작권 메타데이터 기록]
    ↓
[Claude Design: 배경 제거·크롭·디테일 보정 요청]
    ↓
[앱 자산으로 사용 + Credits에 자동 등록]
```

### 5.5 한계 인식

- Anthropic은 제품 한계에 대해 투명하다. 디자인 시스템 임포트는 깔끔한 코드베이스에서 가장 잘 작동하며, 지저분한 소스 코드는 지저분한 출력을 낳는다. 협업 기능은 기본적이며 아직 완전한 멀티플레이어가 아니다. 편집 경험도 거친 부분이 있다.
- 1인 개발자에겐 멀티플레이어 이슈 무관
- 복잡한 애니메이션 제작은 여전히 Lottie/Rive 필요 — Claude Design이 다 해주진 않음
- 리서치 프리뷰 단계라 API 불안정 가능성 → Git에 디자인 파일 백업 필수
